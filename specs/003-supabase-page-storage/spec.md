# Feature Specification: 페이지 서버 저장 및 소유자별 접근 제어

**Feature Branch**: `worktree-wt1`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "페이지 게시글 기능 구현할거야. 이미 기존에 로컬 스토리지로 구현 되어 있었는데, 내가 supabase에 만든 Page 테이블에 저장되도록 변경해줘. 내가 만든 DB 테이블 구조는 변경하면 안 돼. 로그인 된 유저만 등록 가능하고, 자신의 글만 조회 가능하고, 자신의 글만 삭제할 수 있어야 돼."

## Clarifications

### Session 2026-07-16

- Q: 페이지 수정 내용을 언제 서버에 저장할까? → A: 입력이 멈추고 약 800ms 후 저장(디바운스). 편집기를 벗어나거나 다른 페이지로 이동하면 즉시 저장
- Q: 새 페이지는 언제 서버에 만들어질까? → A: 새 페이지 버튼을 누른 즉시 서버에 빈 페이지를 만든다. 아무것도 입력하지 않고 벗어나면 그 빈 페이지는 자동 삭제
- Q: 목록을 불러오는 동안 무엇을 보여줄까? → A: 목록 자리에 스켈레톤을 표시하고, 빈 상태 안내는 로딩이 끝난 뒤에만 노출
- Q: 저장·삭제가 실패하면 화면을 어떻게 되돌릴까? → A: 화면에 먼저 즉시 반영하고, 실패하면 이전 상태로 되돌린 뒤 실패를 안내(낙관적 갱신 + 롤백)
- Q: 이 개념의 표준 용어를 무엇으로 할까? → A: "페이지(Page)"로 통일. 저장소 테이블·슬래시 명령·스펙·코드 식별자를 한 단어로 정렬하고 "게시글/Post"는 쓰지 않는다

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 내 페이지가 계정에 저장된다 (Priority: P1)

로그인한 사용자가 페이지를 작성하면 그 페이지가 브라우저가 아니라 사용자의 계정에 귀속되어 보관된다. 다른 브라우저나 기기에서 같은 계정으로 로그인해도 자신이 쓴 페이지가 그대로 보인다. 브라우저 데이터를 지워도 페이지는 사라지지 않는다.

**Why this priority**: 이 기능의 존재 이유다. 지금은 페이지가 브라우저에만 있어서 기기를 바꾸면 사라진다. 이 스토리 하나만 구현해도 "페이지가 안전하게 보관된다"는 핵심 가치가 전달된다.

**Independent Test**: A 계정으로 로그인해 페이지를 작성하고, 브라우저 저장소를 비운 뒤 같은 계정으로 다시 로그인했을 때 페이지가 그대로 보이면 통과다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 페이지 목록을 보고 있을 때, **When** 새 페이지를 만들고 제목과 내용을 입력하면, **Then** 그 페이지가 사용자의 계정에 보관되고 목록에 나타난다.
2. **Given** 사용자가 페이지를 작성한 상태에서, **When** 브라우저 저장소를 비우고 같은 계정으로 다시 로그인하면, **Then** 작성했던 페이지가 모두 그대로 보인다.
3. **Given** 사용자가 기존 페이지를 열어 둔 상태에서, **When** 제목이나 내용을 수정하면, **Then** 변경 내용이 계정에 보관되고 다시 접속해도 유지된다.
4. **Given** 사용자가 페이지를 작성 중일 때, **When** 저장이 실패하면, **Then** 사용자는 저장되지 않았음을 알리는 메시지를 보고, 입력한 내용은 편집기에 그대로 남아 다시 저장을 시도할 수 있다.

---

### User Story 2 - 남의 페이지는 보이지 않는다 (Priority: P1)

각 사용자는 자신이 작성한 페이지만 볼 수 있다. 다른 사용자의 페이지는 목록에도 나타나지 않고, 직접 접근을 시도해도 열람할 수 없다.

**Why this priority**: 개인 노트 제품에서 남의 페이지가 보이는 것은 즉시 신뢰를 무너뜨리는 결함이다. 저장(스토리 1)과 분리해서 미룰 수 없다 — 저장이 되는 순간부터 격리가 성립해야 한다.

**Independent Test**: A 계정으로 페이지를 작성한 뒤 B 계정으로 로그인해 목록이 비어 있고 A의 페이지에 접근할 수 없으면 통과다.

**Acceptance Scenarios**:

1. **Given** A 계정이 페이지를 작성해 둔 상태에서, **When** B 계정으로 로그인해 페이지 목록을 보면, **Then** A의 페이지는 하나도 보이지 않는다.
2. **Given** B 계정으로 로그인한 상태에서, **When** A의 페이지를 직접 지정해 열람을 시도하면, **Then** 열람되지 않는다.
3. **Given** 로그아웃 상태에서, **When** 페이지 목록 화면에 접근하면, **Then** 어떤 사용자의 페이지도 보이지 않는다.

---

### User Story 3 - 로그인해야 페이지를 쓸 수 있다 (Priority: P1)

로그인하지 않은 사람은 페이지를 만들 수 없다. 페이지 작성은 로그인한 사용자에게만 허용되며, 작성된 페이지는 작성자 본인에게 귀속된다.

**Why this priority**: 페이지에 소유자가 없으면 스토리 1·2의 격리가 성립하지 않는다. 저장 규칙의 전제 조건이다.

**Independent Test**: 로그아웃 상태에서 페이지 작성을 시도했을 때 페이지가 생성되지 않고 로그인을 요구받으면 통과다.

**Acceptance Scenarios**:

1. **Given** 로그아웃 상태에서, **When** 페이지 작성을 시도하면, **Then** 페이지가 만들어지지 않고 로그인이 필요하다는 안내를 받는다.
2. **Given** 로그인한 사용자가, **When** 페이지를 만들면, **Then** 그 페이지의 소유자는 본인으로 기록된다.
3. **Given** 사용자가 페이지 작성 중 세션이 만료되면, **When** 저장이 시도되면, **Then** 저장이 거부되고 사용자는 다시 로그인하라는 안내를 받는다.

---

### User Story 4 - 내 페이지만 삭제할 수 있다 (Priority: P2)

사용자는 자신의 페이지를 삭제할 수 있다. 삭제는 되돌릴 수 없으므로 실행 전에 확인을 거친다. 다른 사용자의 페이지는 삭제할 수 없다.

**Why this priority**: 가치 있는 기능이지만 저장·격리(P1)가 동작한 뒤에 얹을 수 있는 독립 슬라이스다.

**Independent Test**: A 계정으로 자신의 페이지를 삭제하면 목록에서 사라지고 다시 로그인해도 없으며, B 계정이 A의 페이지 삭제를 시도하면 실패하고 A의 페이지가 그대로 남아 있으면 통과다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 자신의 페이지 목록을 보고 있을 때, **When** 페이지 삭제를 실행하고 확인하면, **Then** 그 페이지는 목록에서 사라지고 다시 접속해도 보이지 않는다.
2. **Given** 사용자가 삭제 확인을 요청받았을 때, **When** 확인을 취소하면, **Then** 페이지는 삭제되지 않고 목록에 그대로 남는다.
3. **Given** B 계정으로 로그인한 상태에서, **When** A의 페이지 삭제를 시도하면, **Then** 삭제되지 않고 A의 페이지는 그대로 유지된다.
4. **Given** 사용자가 삭제를 확인해 페이지가 목록에서 사라진 직후, **When** 서버에서 삭제가 실패하면, **Then** 그 페이지는 목록에 다시 나타나고 사용자는 삭제되지 않았음을 안내받는다.

---

### Edge Cases

- 새 사용자가 처음 로그인하면 페이지가 하나도 없으므로, 불러오기가 끝난 뒤 목록 화면은 빈 상태 안내를 보여준다.
- 페이지가 있는 사용자가 로그인할 때 불러오는 동안에는 스켈레톤만 보이며, 빈 상태 안내가 잠깐이라도 스쳐 지나가지 않는다.
- 목록 불러오기가 실패하면 사용자는 실패를 안내받으며, 이 경우에도 빈 상태 안내("페이지가 없습니다")를 대신 보여주지 않는다. 페이지가 없는 것과 불러오지 못한 것은 구분해서 알린다.
- 삭제한 페이지는 복원할 수 없다. 사용자가 삭제를 실행하면 확인 절차를 거친 뒤에만 사라진다.
- 이전 방식으로 브라우저에 페이지가 남아 있는 사용자가 로그인하면, 그 페이지는 목록에 나타나지 않고 서버에 보관된 페이지만 보인다.
- 네트워크가 끊긴 상태에서 페이지를 작성·수정·삭제하면 사용자는 실패를 명확히 안내받고 작성 중이던 내용을 잃지 않는다.
- 새 페이지를 만들면 제목·내용이 빈 상태로 편집기가 열리며, 그동안 목록에서는 "제목 없음"으로 식별된다. 아무것도 입력하지 않고 벗어나면 그 페이지는 자동으로 사라진다(FR-019).
- 제목만 쓰고 내용을 비워 두거나 그 반대인 페이지는 정상적인 페이지로 보관된다. 자동 삭제 대상은 제목과 내용이 모두 빈 경우로 한정한다.
- 같은 계정으로 두 탭을 열어 각각 다른 페이지를 수정하면 각 페이지의 최신 저장 내용이 유지된다.
- 같은 페이지를 두 탭에서 동시에 수정하면 나중에 저장된 내용이 최종 상태가 된다.
- 페이지 작성 도중 세션이 만료되면 저장이 거부되고 재로그인을 안내받는다.
- 매우 긴 내용(수만 자)도 저장되고 다시 열었을 때 손실 없이 보인다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자가 작성한 페이지를 브라우저가 아닌 서버에 보관해야 하며, 같은 계정으로 접속하면 기기·브라우저와 무관하게 동일한 페이지 목록을 제공해야 한다.
- **FR-002**: 시스템은 로그인한 사용자에게만 페이지 생성을 허용해야 한다.
- **FR-003**: 시스템은 생성되는 모든 페이지에 작성자를 소유자로 기록해야 하며, 소유자 없는 페이지는 존재할 수 없다.
- **FR-004**: 시스템은 사용자에게 자신이 소유한 페이지만 조회 결과로 제공해야 한다.
- **FR-005**: 시스템은 사용자가 자신이 소유한 페이지만 삭제할 수 있도록 해야 한다.
- **FR-006**: 시스템은 사용자가 자신이 소유한 페이지의 제목과 내용을 수정할 수 있도록 해야 하며, 수정 결과를 서버에 보관해야 한다.
- **FR-007**: 시스템은 소유자가 아닌 사용자의 조회·수정·삭제 시도를 거부해야 하며, 이 거부는 클라이언트 화면 로직이 아니라 데이터 저장소 자체에서 강제되어야 한다.
- **FR-008**: 시스템은 저장·조회·삭제가 실패했을 때 사용자에게 실패 사실을 알려야 하며, 성공한 것처럼 보이게 해서는 안 된다.
- **FR-009**: 시스템은 기존에 확정된 페이지 저장 구조를 변경하지 않고 사용해야 한다. 저장 가능한 페이지 정보는 식별자·생성 시각·제목·내용·소유자로 한정된다.
- **FR-010**: 시스템은 로그아웃 시 이전 사용자의 페이지가 화면에 남지 않도록 해야 한다.
- **FR-011**: 시스템은 페이지 목록을 최신 페이지가 먼저 보이도록 정렬해 제공해야 한다.
- **FR-012**: 시스템은 즐겨찾기 기능과 휴지통(삭제 후 복원) 기능을 제공하지 않아야 한다. 페이지 목록 탐색에서 즐겨찾기·휴지통 항목과 관련 조작(즐겨찾기 지정·해제, 휴지통 이동, 복원)을 제거한다. 삭제는 즉시 영구 삭제이며 복원 경로는 존재하지 않는다.
- **FR-013**: 시스템은 페이지의 "수정 시각" 표시와 최근 수정순 정렬을 제공하지 않아야 한다. 목록의 시각 표시와 정렬은 생성 시각만을 기준으로 한다.
- **FR-014**: 시스템은 첫 로그인 시 예시 페이지를 자동 생성하지 않아야 하며, 이전 방식으로 브라우저에 저장되어 있던 페이지를 서버로 옮기지 않아야 한다. 새 사용자는 빈 목록에서 시작한다. 브라우저에 남아 있는 이전 페이지 데이터는 정리해 화면에 영향을 주지 않도록 한다.
- **FR-015**: 삭제는 되돌릴 수 없으므로, 시스템은 삭제를 실행하기 전에 사용자에게 확인을 받아야 한다.
- **FR-016**: 시스템은 페이지 수정 내용을 별도의 저장 조작 없이 자동으로 보관해야 한다. 저장은 사용자의 입력이 멈춘 뒤 약 800ms 후에 실행하며, 입력이 계속되는 동안에는 저장을 미뤄 연속된 타이핑이 한 번의 저장으로 합쳐지도록 한다.
- **FR-017**: 시스템은 사용자가 편집을 중단하고 다른 페이지로 이동하거나 편집기를 벗어날 때, 아직 보관되지 않은 변경이 있으면 지연을 기다리지 않고 즉시 보관해야 한다.
- **FR-018**: 시스템은 사용자가 새 페이지 만들기를 실행한 즉시 제목과 내용이 빈 페이지를 소유자와 함께 보관하고, 그 페이지의 편집기를 열어야 한다.
- **FR-019**: 시스템은 새로 만든 페이지가 제목과 내용이 모두 빈 채로 남은 상태에서 사용자가 그 페이지를 벗어나면, 그 페이지를 자동으로 삭제해야 한다. 이 자동 삭제에는 FR-015의 삭제 확인을 요구하지 않는다.
- **FR-020**: 시스템은 페이지 목록을 불러오는 동안 목록 자리에 자리표시(스켈레톤)를 보여줘야 하며, 불러오기가 끝나면 실제 목록으로 교체해야 한다.
- **FR-021**: 시스템은 "페이지가 없습니다" 빈 상태 안내를 불러오기가 완료되어 페이지가 0건임이 확정된 뒤에만 보여줘야 한다. 불러오는 중에는 빈 상태 안내를 보여서는 안 된다.
- **FR-022**: 시스템은 페이지 생성·수정·삭제를 화면에 먼저 반영해 사용자가 서버 응답을 기다리지 않도록 해야 한다.
- **FR-023**: 시스템은 페이지 생성 또는 삭제를 화면에 먼저 반영한 뒤 서버에서 실패하면, 그 조작 직전의 상태로 화면을 되돌려야 한다(생성 실패 시 목록에서 제거, 삭제 실패 시 목록에 복원). 되돌린 사실과 실패 원인을 사용자에게 알려야 하며, 화면 상태와 서버 상태가 어긋난 채로 남아서는 안 된다.
- **FR-024**: 시스템은 페이지 수정 내용의 저장이 실패했을 때 사용자가 입력한 내용을 되돌려서는 안 된다. 입력한 내용은 편집기에 그대로 유지하고, 아직 보관되지 않았음을 사용자에게 알리며, 이후 입력이나 재시도로 다시 저장을 시도할 수 있어야 한다.

### Key Entities

- **페이지(Page)**: 사용자가 작성한 하나의 문서(이전에는 "글", "게시글", 코드에서는 `Post`로도 불렸다). 고유 식별자, 생성 시각, 제목, 내용, 소유자를 가진다. 제목과 내용은 비어 있을 수 있다. 소유자는 반드시 존재하며 가입된 사용자 한 명을 가리킨다. 이 구조는 이미 확정되어 있으며 이번 작업에서 변경하지 않는다.
- **사용자(User)**: 로그인한 사람. 페이지의 소유자가 되며, 자신이 소유한 페이지에만 접근할 수 있다. 기존 인증 체계를 그대로 사용한다.

**표준 용어**: 이 개념의 표준 명칭은 "페이지(Page)"다. 저장소 테이블명, 사용자에게 노출되는 슬래시 명령(`/page`), 스펙 서술, 코드 식별자가 모두 이 한 단어를 쓴다. "글", "게시글", `Post`는 사용하지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 한 기기에서 작성한 페이지가 다른 기기에서 같은 계정으로 로그인했을 때 100% 그대로 보인다.
- **SC-002**: 브라우저 저장소를 완전히 비운 뒤 재로그인해도 작성한 페이지가 하나도 유실되지 않는다.
- **SC-003**: 다른 사용자의 페이지가 목록·검색·직접 접근 어떤 경로로도 노출되는 사례가 0건이다.
- **SC-004**: 로그인하지 않은 상태에서 페이지가 생성되는 사례가 0건이다.
- **SC-005**: 다른 사용자의 페이지가 삭제되는 사례가 0건이다.
- **SC-006**: 페이지 목록이 로그인 후 2초 이내에 표시된다.
- **SC-007**: 페이지를 저장한 뒤 새로고침했을 때 방금 저장한 내용이 그대로 보이는 비율이 100%다.
- **SC-008**: 저장·삭제 실패 시 사용자가 실패 사실을 인지할 수 있는 안내가 100% 제공된다.
- **SC-009**: 페이지가 있는 사용자에게 빈 상태 안내가 잘못 노출되는 사례가 0건이다.
- **SC-010**: 사용자가 연속으로 타이핑하는 동안 저장 요청이 반복 발생하지 않으며, 타이핑을 멈춘 뒤 1초 이내에 변경 내용이 보관된다.

## Assumptions

- 기존 Google 로그인 인증 체계를 그대로 사용하며, 이번 작업에서 인증 방식은 바꾸지 않는다.
- 페이지의 소유자 판별은 로그인 세션의 사용자 신원을 기준으로 한다.
- 소유자별 접근 제한은 데이터 저장소 수준에서 강제한다. 확정된 저장 구조(칼럼 구성)는 변경하지 않지만, 현재 이 저장소에는 접근 규칙이 하나도 정의되어 있지 않아 어떤 읽기·쓰기도 허용되지 않는 상태다. 따라서 요구된 소유자별 권한(본인 페이지만 조회·생성·수정·삭제)을 성립시키기 위한 접근 규칙 추가는 이번 작업 범위에 포함한다. 이는 칼럼 구성 변경이 아니다.
- 페이지 목록의 기본 정렬 기준은 생성 시각 최신순으로 한다.
- 페이지 수정은 자동 저장 방식이며 별도의 저장 버튼을 도입하지 않는다. 저장 시점은 FR-016·FR-017에 정의한다. 800ms라는 지연 값은 타이핑을 끊기게 하지 않으면서 요청을 충분히 합치는 값으로 채택했으며, 사용자에게 노출되는 설정이 아니다.
- 동시 수정 충돌 해결(잠금, 병합, 버전 관리)은 이번 범위에서 제외한다. 나중에 저장된 내용이 최종 상태가 된다.
- 오프라인 작성 큐잉·동기화는 이번 범위에서 제외한다.
- 페이지 공유·협업·공개 기능은 이번 범위에서 제외한다. 모든 페이지는 비공개다.
- 사용자 프로필은 이미 서버에 보관되고 있으며 이번 범위에서 변경하지 않는다.
- 즐겨찾기·휴지통·수정 시각은 확정된 저장 구조에 담을 자리가 없어 제품에서 제거하기로 결정했다(FR-012, FR-013). 이는 기능 축소이며, 기존 사용자가 즐겨찾기로 지정했거나 휴지통에 두었던 상태는 보존되지 않는다.
- 삭제는 되돌릴 수 없다. 휴지통이라는 안전망이 사라지므로 실수 삭제를 막기 위해 삭제 전 확인 절차를 둔다(FR-015).
- 예시 페이지를 만들지 않으므로 새 사용자는 빈 목록을 처음 보게 된다. 기존 빈 상태 안내 화면이 이 경우를 담당한다.
- 표준 용어를 "페이지(Page)"로 통일하기로 했으므로, 이 개념을 다른 이름으로 부르던 기존 부분의 명칭 정렬이 이번 작업에 포함된다. 이는 사용자에게 보이는 동작을 바꾸지 않는 명칭 변경이며, 확정된 저장 구조는 그대로 둔다.
