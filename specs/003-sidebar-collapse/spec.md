# Feature Specification: 사이드바 접기/펼치기

**Feature Branch**: `worktree-wt3`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "사이드바에 접기/펼치기 기능을 구현해줘. 사이드바에 버튼을 하나 만들고, 이걸 클릭하면 접기, 다시 클릭하면 펼치기가 되도록 해줘."

## Clarifications

### Session 2026-07-16

- Q: 같은 브라우저에서 계정을 전환하면 접힘 상태가 공유되어야 하나? → A: 브라우저 단위 하나로 공유한다(계정과 무관). 접힘 상태는 계정 설정이 아니라 기기·브라우저 설정이다.
- Q: 접힌 상태에서 검색 때문에 자동으로 펼쳐졌을 때, 그 펼침이 저장된 선호가 되나? → A: 된다. 자동 펼침은 사용자가 토글 버튼을 누른 것과 동일하게 취급하며 그대로 저장된다. 일시적 펼침 상태는 두지 않는다.
- Q: 토글 버튼은 항상 보여야 하나, 호버 시에만 나타나야 하나? → A: 두 상태 모두에서 항상 보인다. 호버로만 드러나는 방식은 채택하지 않는다.
- Q: 측정 수단이 없는 SC-004(처음 보는 사용자 90% 발견률)를 어떻게 하나? → A: 사용성 테스트 없이 검증 가능한 발견 가능성 기준으로 교체한다(스크롤 없이 보임·상시 노출·접근 가능한 이름·탭 도달).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사이드바를 접어 읽기 공간 확보하기 (Priority: P1)

글을 읽거나 쓰는 데 집중하려는 사용자가 사이드바의 토글 버튼을 클릭한다. 사이드바가 아이콘만 남는 좁은 레일로 접히면서 확보된 가로 공간이 본문 영역으로 넘어간다. 접힌 상태에서도 내비게이션 아이콘은 남아 있어 이동이 막히지 않는다. 다시 버튼을 클릭하면 사이드바가 원래 모습으로 돌아온다. 사용자는 내비게이션이 필요할 때와 본문에 집중할 때를 오가며 화면을 자기 작업에 맞춘다.

**Why this priority**: 이 하나만 구현해도 사용자가 요청한 가치("클릭하면 접기, 다시 클릭하면 펼치기")가 온전히 전달된다. 나머지 스토리는 이 토글이 없으면 성립하지 않는다. 이것이 MVP다.

**Independent Test**: 워크스페이스 화면에서 토글 버튼을 클릭해 사이드바가 접히는지, 한 번 더 클릭해 펼쳐지는지, 접힌 동안 본문 영역이 실제로 넓어지는지를 확인하면 단독으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사이드바가 펼쳐진 기본 상태의 워크스페이스에서, **When** 사용자가 토글 버튼을 클릭하면, **Then** 사이드바가 접힌 상태로 바뀌고 본문 영역이 그만큼 넓어진다.
2. **Given** 사이드바가 접힌 상태에서, **When** 사용자가 토글 버튼을 다시 클릭하면, **Then** 사이드바가 접기 이전과 동일한 모습으로 복원된다.
3. **Given** 사이드바가 접힌 상태에서, **When** 사용자가 화면을 보면, **Then** 좁은 레일 안에 토글 버튼이 그대로 보여 다시 펼칠 수 있다.
4. **Given** 사이드바가 접힌 상태에서, **When** 사용자가 내비게이션 아이콘을 선택하면, **Then** 펼쳐진 상태와 동일하게 해당 목록으로 이동하며 사이드바는 접힌 채로 유지된다.
5. **Given** 사이드바가 접힌 상태에서, **When** 사용자가 검색을 시도하면, **Then** 사이드바가 펼쳐지고 검색 입력을 바로 사용할 수 있으며, 검색을 마쳐도 다시 접히지 않고 펼침 상태로 남는다.
6. **Given** 어느 쪽 상태이든, **When** 상태가 전환되면, **Then** 전환이 끊김 없이 보이고 열려 있던 본문 내용이 유실되거나 초기화되지 않는다.

### User Story 2 - 선택한 사이드바 상태가 유지되기 (Priority: P2)

사이드바를 접어둔 사용자가 글을 열고 닫고, 마이 페이지에 다녀오고, 브라우저를 새로고침한다. 사이드바는 매번 다시 접을 필요 없이 사용자가 마지막으로 선택한 상태 그대로 유지된다.

**Why this priority**: 토글 자체(P1)만으로도 기능은 동작하지만, 이동할 때마다 상태가 초기화되면 사용자는 같은 클릭을 반복하게 되어 기능의 가치가 크게 깎인다. 다만 P1 없이는 존재할 수 없으므로 P2다.

**Independent Test**: 사이드바를 접은 뒤 새로고침하고 화면을 이동해도 접힌 상태가 유지되는지 확인하면 단독으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 사이드바를 접어둔 상태에서, **When** 브라우저를 새로고침하면, **Then** 사이드바는 여전히 접힌 상태로 표시된다.
2. **Given** 사용자가 사이드바를 접어둔 상태에서, **When** 사이드바가 있는 다른 화면으로 이동하면, **Then** 그 화면에서도 사이드바는 접힌 상태로 표시된다.
3. **Given** 저장된 상태가 없는 첫 방문 사용자가, **When** 워크스페이스를 열면, **Then** 사이드바는 펼쳐진 기본 상태로 표시된다.
4. **Given** 저장된 상태 값이 없거나 읽을 수 없는 상황에서, **When** 화면이 로드되면, **Then** 오류 없이 펼쳐진 기본 상태로 표시된다.

---

### User Story 3 - 키보드·보조기술로 토글 사용하기 (Priority: P3)

마우스를 쓰지 않거나 스크린 리더를 사용하는 사용자가 키보드만으로 토글 버튼에 도달해 실행하고, 사이드바가 지금 접혀 있는지 펼쳐져 있는지를 인지한다.

**Why this priority**: 접근성은 품질 기준이지만 P1의 토글이 존재한 뒤에 그 위에 얹히는 층이다. 별도 슬라이스로 두어 P1을 막지 않으면서도 누락되지 않게 한다.

**Independent Test**: 키보드만으로 토글에 포커스를 옮겨 실행하고, 버튼이 자신의 현재 상태와 목적을 보조기술에 노출하는지 확인하면 단독으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 워크스페이스 화면에서, **When** 사용자가 키보드로 토글 버튼까지 이동하면, **Then** 포커스 위치가 눈에 보이게 표시된다.
2. **Given** 토글 버튼에 포커스가 있을 때, **When** 사용자가 키보드로 실행하면, **Then** 마우스 클릭과 동일하게 상태가 전환된다.
3. **Given** 보조기술을 사용하는 사용자가, **When** 토글 버튼을 만나면, **Then** 버튼의 목적과 현재 펼침/접힘 상태를 인지할 수 있다.
4. **Given** 움직임을 줄이도록 설정한 사용자가, **When** 상태를 전환하면, **Then** 전환 애니메이션이 생략되고 결과 상태로 즉시 바뀐다.

---

### Edge Cases

- 접힌 상태에서 사용자가 검색을 시도하면? → 사이드바를 펼치고 검색을 사용 가능하게 하며, 그 펼침은 저장되어 유지된다(FR-015, FR-015a).
- 접힌 상태에서 내비 항목을 쓰려 하면? → 아이콘으로 그대로 선택 가능하다(FR-013).
- 접힌 상태에서 카운트 배지나 사용자 이름 같은 텍스트 정보가 필요하면? → 라벨은 감춰지되 호버·포커스 시 확인할 수 있고, 펼치면 전부 돌아온다(FR-014).
- 좁은 화면(≤1024px)에서 접기를 실행하면? 이미 축소된 레일 폭(220px) 규격과 충돌하지 않는가?
- 저장된 상태 값이 손상되었거나 예상치 못한 형식일 때 화면이 깨지지 않는가? → 펼침 기본값으로 복구한다(FR-007).
- 사이드바가 접힌 채로 창 크기를 바꾸면 접힘 상태가 유지되는가?
- 화면 로드 직후 저장된 상태를 읽어오는 사이에 사이드바가 순간 튀어 보이지 않는가?(FR-016)
- 같은 사용자가 두 탭을 열어 한쪽에서 접으면 다른 탭은 어떻게 되는가? → Assumptions 참고(탭 간 실시간 동기화는 범위 밖).
- 아이콘 전용 레일의 최소 폭이 터치·클릭 대상 크기를 만족하는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사이드바에는 접힘/펼침을 전환하는 조작 수단이 하나 있어야 하며, 같은 조작을 반복하면 두 상태를 번갈아 오가야 한다(누를 때마다 접힘 → 펼침 → 접힘).
- **FR-001a**: 토글 조작 수단은 접힘·펼침 두 상태 모두에서 항상 보여야 한다. 마우스를 올렸을 때만 드러나서는 안 되며, 호버가 없는 터치 기기에서도 동일하게 보이고 동작해야 한다.
- **FR-002**: 접힌 상태에서 사이드바는 화면에서 사라지지 않고, 아이콘만 남는 좁은 레일로 축소되어야 한다. 레일 자체는 접힘/펼침 두 상태 모두에서 항상 화면에 존재한다.
- **FR-003**: 사이드바가 접히면 그로 인해 확보된 가로 공간은 본문 영역이 사용해야 하며, 본문 콘텐츠가 잘리거나 겹쳐서는 안 된다.
- **FR-004**: 접힌 상태에서도 사용자가 사이드바를 다시 펼칠 수 있는 조작 수단이 항상 화면에 보여야 한다. 사용자가 되돌아갈 수 없는 상태에 갇혀서는 안 된다.
- **FR-005**: 상태 전환은 사용자가 원인과 결과를 인지할 수 있도록 시각적으로 연속적이어야 하며, 화면 전체가 다시 그려지는 것처럼 보여서는 안 된다.
- **FR-006**: 사용자가 선택한 사이드바 상태는 저장된 환경설정으로 유지되어야 하며, 화면 이동과 브라우저 재방문 이후에도 마지막 선택이 복원되어야 한다.
- **FR-007**: 저장된 상태가 없거나 읽을 수 없으면 시스템은 오류 없이 펼쳐진 상태를 기본값으로 사용해야 한다.
- **FR-008**: 사이드바가 표시되는 모든 화면에서 접힘/펼침 동작과 현재 상태가 일관되게 적용되어야 한다.
- **FR-009**: 토글 조작 수단은 키보드만으로 도달·실행할 수 있어야 하고, 포커스 위치가 시각적으로 드러나야 한다.
- **FR-010**: 토글 조작 수단은 자신의 목적과 현재 접힘/펼침 상태를 보조기술에 노출해야 한다.
- **FR-011**: 상태 전환은 좁은 화면(≤1024px)을 포함한 지원 화면 폭 전체에서 동작해야 하며, 각 화면 폭의 기존 레이아웃 규격을 깨뜨리지 않아야 한다.
- **FR-012**: 움직임을 줄이도록 설정한 사용자에게는 전환 애니메이션을 생략하고 결과 상태를 즉시 적용해야 한다.
- **FR-013**: 접힌 상태에서도 모든 내비게이션 항목(전체 글·즐겨찾기·최근 항목·휴지통)은 아이콘 형태로 남아 계속 선택할 수 있어야 하며, 현재 선택된 항목이 어느 것인지 접힌 상태에서도 구분되어야 한다.
- **FR-014**: 접힌 상태에서는 텍스트 요소(브랜드 이름, 섹션 라벨, 내비 라벨, 카운트 배지, 사용자 이름)를 감춘다. 감춰진 각 아이콘의 의미는 마우스를 올리거나 키보드로 포커스했을 때 라벨로 확인할 수 있어야 한다.
- **FR-015**: 접힌 상태에서 사용자가 검색을 시도하면 시스템은 사이드바를 펼치고 검색 입력을 바로 사용할 수 있는 상태로 만들어야 한다. 좁은 레일에서 검색어를 입력하도록 요구해서는 안 된다.
- **FR-015a**: FR-015의 자동 펼침은 사용자가 토글 버튼을 눌러 펼친 것과 동일하게 취급되어야 한다. 즉 펼침 상태가 그대로 저장되며(FR-006), 검색이 끝나도 자동으로 다시 접히지 않는다. 저장되지 않는 일시적 펼침 상태는 존재하지 않는다.
- **FR-016**: 저장된 상태를 읽어오는 동안 사이드바가 펼침에서 접힘으로(또는 그 반대로) 순간적으로 튀어 보여서는 안 된다.
- **FR-017**: 접힘 상태는 로그인 계정과 무관하게 기기·브라우저 단위로 보관되어야 한다. 로그아웃·계정 전환·재로그인은 접힘 상태를 초기화하지 않으며, 같은 브라우저의 다음 사용자도 마지막 선택을 이어받는다.

### Key Entities

- **사이드바 표시 상태(Sidebar display preference)**: 사이드바가 현재 접혀 있는지 펼쳐져 있는지를 나타내는 기기·브라우저 단위 UI 환경설정. 두 값만 가진다(접힘/펼침). 기본값은 펼침. 로그인 계정에 종속되지 않으며, 사용자가 명시적으로 토글할 때만 바뀐다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 단 한 번의 조작으로 사이드바를 접거나 펼칠 수 있다.
- **SC-002**: 사이드바를 접으면 본문에 쓸 수 있는 가로 공간이 즉시 넓어지며, 1280px 폭 화면 기준 최소 200px 이상 늘어난다.
- **SC-003**: 토글 조작 후 결과 상태가 화면에 반영되기까지 0.3초를 넘지 않아, 사용자가 기다린다고 느끼지 않는다.
- **SC-004**: 토글 조작 수단은 안내 없이 발견할 수 있다 — 워크스페이스에 들어선 첫 화면에서 스크롤 없이 보이고, 호버 없이 항상 노출되며, 접근 가능한 이름을 가지고, 화면 진입 후 키보드 탭 5회 이내로 도달할 수 있다. 네 조건 모두 실제 화면에서 확인 가능하다.
- **SC-005**: 접힌 상태에서 다시 펼치지 못하고 갇히는 경로가 0건이다.
- **SC-006**: 마우스를 전혀 쓰지 않고 키보드만으로 접기와 펼치기를 모두 수행할 수 있다.
- **SC-007**: 사이드바 상태 전환이 열려 있던 글의 내용이나 편집 중이던 입력을 유실시키는 경우가 0건이다.

## Assumptions

- 이 기능은 사이드바(좌측 레일)에만 적용되며, 리스트 페인이나 상세 영역의 접기는 이번 범위가 아니다.
- 접힘 상태는 계정 설정이 아니라 기기·브라우저 단위 설정이다. 같은 브라우저를 쓰는 한 로그인 계정이 바뀌어도 마지막 선택이 그대로 유지되며, 다른 기기·브라우저와는 공유되지 않는다.
- 토글 버튼은 접힘/펼침 두 상태 모두에서 사이드바 안에 있다. 아이콘 전용 레일을 택했으므로 버튼이 레일 바깥으로 나갈 필요가 없고, "사이드바에 버튼을 하나 만들어달라"는 요청을 그대로 만족한다.
- 기본 상태는 펼침이다. 기존 사용자에게 이 기능이 추가되어도 지금과 동일한 첫 화면을 보게 된다.
- 사이드바가 표시되는 화면은 현재 워크스페이스와 마이 페이지 두 곳이며, 두 화면이 동일한 사이드바를 공유하는 현재 구조를 전제한다.
- 접힘 상태는 기존에 글 목록을 저장하는 것과 동일한 브라우저 로컬 저장 방식을 따른다. 서버나 사용자 계정에 저장하지 않는다(로그인 없이도 동작하며 운영 비용이 늘지 않는다).
- 여러 탭을 동시에 열었을 때 한 탭의 변경이 다른 탭에 실시간 반영되는 동기화는 범위가 아니다. 각 탭은 로드 시점의 저장된 값을 사용한다.
- 화면 폭에 따른 자동 접기는 이번 범위가 아니다. 접기는 사용자의 명시적 조작으로만 발생한다.
- 별도의 키보드 단축키는 이번 범위가 아니다. 버튼 조작과 표준 키보드 접근(포커스 후 실행)만 제공한다.
- 시각 표현(레일 폭·아이콘·모션 값)은 기존 디자인 시스템의 토큰과 스케일을 따르며, 새로운 임의 값을 도입하지 않는다. 접힌 레일 폭과 폭 전환 모션은 현재 토큰에 없으므로 기존 스케일과 일관되게 추가한다.
