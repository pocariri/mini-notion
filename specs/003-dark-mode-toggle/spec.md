## Clarifications

### Session 2026-07-16

- Q: 사이드바가 두 개(업무 화면 / 마이 페이지)인데 테마 토글을 어디에 둘까? → A: 두 사이드바 모두에 둔다.
- Q: 마이 페이지의 "모든 데이터 초기화"가 테마 설정도 초기화하나? → A: 아니다. 테마는 기기 설정이므로 초기화 대상이 아니며 그대로 유지된다.
- Q: 테마를 전환하는 순간이 어떻게 보여야 하나? → A: 즉시 전환한다. 전환 순간에만 색 트랜지션을 억제해 배경과 컨트롤이 동시에 바뀌게 하고, hover 등 평소 트랜지션은 그대로 둔다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사이드바 토글로 다크모드 켜고 끄기 (Priority: P1)

업무 화면에서 일하던 사용자가 사이드바에 있는 테마 토글을 누른다. 화면 전체 — 사이드바, 목록, 에디터 — 가 즉시 어두운 테마로 바뀐다. 다시 누르면 원래의 밝은 테마로 돌아온다. 작성 중이던 글이나 스크롤 위치, 선택한 메뉴는 그대로 유지된다.

**Why this priority**: 사용자가 요청한 기능의 핵심이자 그 자체로 완결된 가치다. 저장이나 운영체제 연동이 없어도 "어두운 화면에서 글을 쓸 수 있다"는 목적을 이 스토리만으로 달성한다.

**Independent Test**: 업무 화면에서 토글을 클릭해 전체 UI가 어두워지는지, 다시 클릭해 밝아지는지 확인한다. 전환 중 작업 내용이 손실되지 않아야 한다.

**Acceptance Scenarios**:

1. **Given** 라이트 테마로 업무 화면을 보고 있을 때, **When** 사이드바의 테마 토글을 클릭하면, **Then** 페이지 새로고침 없이 화면 전체가 다크 테마로 바뀐다.
2. **Given** 다크 테마 상태일 때, **When** 토글을 다시 클릭하면, **Then** 라이트 테마로 되돌아간다.
3. **Given** 글을 작성하던 중일 때, **When** 테마를 전환하면, **Then** 입력 중이던 내용·커서·스크롤 위치·선택된 내비 항목이 그대로 유지된다.
4. **Given** 어느 테마든, **When** 토글을 보면, **Then** 현재 어떤 테마인지와 클릭 시 무엇으로 바뀌는지를 아이콘·라벨로 알 수 있다.
5. **Given** 다크 테마 상태일 때, **When** 업무 화면의 모든 요소를 살펴보면, **Then** 밝은 배경이 남아 있는 영역 없이 일관되게 어둡다.
6. **Given** 마이 페이지에 있을 때, **When** 그 화면 사이드바의 테마 토글을 클릭하면, **Then** 업무 화면에서와 똑같이 테마가 전환되고, 업무 화면으로 돌아가도 같은 테마가 유지된다.
7. **Given** 어느 화면에서든 테마를 전환할 때, **When** 전환 순간을 보면, **Then** 배경과 컨트롤이 어긋나 따라오지 않고 한 번에 바뀐다.

---

### User Story 2 - 선택한 테마가 유지됨 (Priority: P2)

다크모드를 켠 사용자가 브라우저를 닫았다가 나중에 다시 들어온다. 매번 다시 켤 필요 없이 처음부터 다크 테마로 열린다. 로딩 순간에 밝은 화면이 번쩍하는 일도 없다.

**Why this priority**: 유지되지 않는 테마 설정은 매 방문마다 다시 누르게 만들어 실질적으로 쓸모가 없다. 다만 US1 없이는 성립하지 않으므로 P2다.

**Independent Test**: 다크모드로 전환한 뒤 새로고침·재방문하여 다크 테마가 유지되는지, 로드 순간 밝은 화면 깜빡임이 없는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 다크 테마를 선택한 상태일 때, **When** 페이지를 새로고침하면, **Then** 다크 테마로 렌더되며 중간에 라이트 테마가 잠깐 보이지 않는다.
2. **Given** 다크 테마를 선택하고 브라우저를 완전히 닫았을 때, **When** 나중에 다시 접속하면, **Then** 다크 테마로 열린다.
3. **Given** 다크 테마를 선택한 상태일 때, **When** 다른 화면(마이 페이지 등)으로 이동하면, **Then** 그 화면도 다크 테마로 표시된다.
4. **Given** 저장된 테마 설정이 없거나 알아볼 수 없는 값일 때, **When** 앱에 접속하면, **Then** 오류 없이 운영체제 설정에 따른 테마로 표시된다.
5. **Given** 다크 테마를 선택하고 로그아웃했을 때, **When** 로그인 화면을 보면, **Then** 로그인 화면도 다크 테마로 표시된다.

---

### User Story 3 - 첫 방문 시 운영체제 설정 존중 (Priority: P3)

한 번도 테마를 고른 적 없는 사용자가 처음 접속한다. 운영체제를 다크 모드로 쓰고 있다면 앱도 다크로 열려 눈부심 없이 시작한다. 이후 사용자가 토글로 직접 고르면 그 선택이 운영체제 설정보다 우선한다.

**Why this priority**: 첫인상을 개선하지만 없어도 기능은 완전히 동작한다. 사용자가 한 번만 토글하면 US2가 이후를 책임진다.

**Independent Test**: 저장된 설정이 없는 상태에서 운영체제를 다크로 두고 접속해 다크로 열리는지, 이후 직접 라이트를 고르면 운영체제가 다크여도 라이트가 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 테마를 고른 적 없고 운영체제가 다크 모드일 때, **When** 앱에 접속하면, **Then** 다크 테마로 열린다.
2. **Given** 테마를 고른 적 없고 운영체제가 라이트 모드일 때, **When** 앱에 접속하면, **Then** 라이트 테마로 열린다.
3. **Given** 사용자가 토글로 라이트를 직접 선택한 상태일 때, **When** 운영체제가 다크 모드로 바뀌면, **Then** 사용자의 선택인 라이트가 그대로 유지된다.
4. **Given** 아직 테마를 고른 적 없는 상태로 앱을 보고 있을 때, **When** 운영체제의 다크 모드 설정이 바뀌면, **Then** 앱도 그에 맞춰 바뀐다.

---

### Edge Cases

- **사이드바가 없는 화면**: `/login`과 스플래시에는 사이드바가 없다. 이 화면들도 저장된 테마를 따라 표시되어야 하지만 토글은 노출되지 않는다. 사용자는 로그인 후 사이드바가 있는 화면에서 테마를 바꾼다.
- **두 토글의 일관성**: 업무 화면과 마이 페이지의 토글은 같은 설정을 가리킨다. 한쪽에서 다크로 바꾸고 다른 화면으로 이동하면 그 화면의 토글도 다크 상태로 표시되어야 한다.
- **로그아웃 상태**: 테마가 기기 단위로 저장되므로(FR-011) 로그인하지 않은 사용자가 보는 로그인 화면과 스플래시도 저장된 테마로 열린다. 로그아웃해도 테마 선택은 초기화되지 않는다.
- **같은 기기, 다른 계정**: 테마는 계정이 아니라 기기에 묶이므로, 한 기기를 여러 계정이 나눠 쓰면 마지막에 선택된 테마를 공유한다. 이는 의도된 동작이다.
- **전체 초기화**: "모든 데이터 초기화"는 글·프로필·세션만 지우고 테마는 남긴다(FR-018). 초기화 후 로그인 화면으로 이동해도 테마가 갑자기 바뀌지 않는다.
- **탭 여러 개**: 같은 앱을 여러 탭에서 열어 둔 상태에서 한 탭의 테마를 바꾸면 다른 탭은 다음 방문·새로고침 시점에 반영한다. 실시간 동기화는 요구하지 않는다.
- **로드 시 깜빡임**: 저장된 테마를 반영하기 전에 다른 테마가 먼저 그려져 화면이 번쩍이는 현상은 허용하지 않는다.
- **저장 실패·손상**: 브라우저 저장소를 쓸 수 없거나(프라이빗 모드·용량 초과) 저장값이 손상된 경우, 앱은 오류 화면 없이 운영체제 설정에 따른 테마로 정상 동작하고 해당 세션 내 전환은 계속 가능하다. 다만 이 경우 선택이 다음 방문까지 유지되지 않을 수 있다.
- **운영체제 설정을 알 수 없는 환경**: 운영체제의 다크 모드 선호를 판별할 수 없으면 라이트로 표시한다.
- **전환 애니메이션**: 테마 전환은 애니메이션 없이 즉시 이뤄지므로(FR-003a) 모션 최소화를 설정한 사용자와 그렇지 않은 사용자가 같은 동작을 본다. 전환 순간의 트랜지션 억제가 hover 등 평소 트랜지션까지 영구히 없애서는 안 된다.
- **외부 이미지**: 글 커버로 쓰이는 외부 고양이 사진은 다크 테마에서도 원본 그대로 표시한다. 이미지 자체의 색은 바꾸지 않는다.
- **의미색 대비**: 다크 테마에서도 즐겨찾기 별(앰버), 삭제(레드), 저장 완료(그린) 같은 의미색이 배경과 구분되어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

**토글 컨트롤**

- **FR-001**: 업무 화면과 마이 페이지의 사이드바 모두 테마를 전환하는 토글 컨트롤을 제공해야 한다. 두 토글은 같은 테마 설정을 읽고 쓰며 동일하게 동작·표시된다.
- **FR-002**: 토글은 현재 활성 테마가 무엇인지 시각적으로 표시해야 한다.
- **FR-003**: 사용자가 토글을 조작하면 시스템은 페이지 새로고침 없이 즉시 테마를 전환해야 한다.
- **FR-003a**: 테마 전환은 애니메이션 없이 이뤄져야 하며, 배경·텍스트·보더·컨트롤이 서로 어긋나지 않고 한 번에 바뀌어야 한다. 전환이 끝난 뒤에는 hover 같은 평소 상호작용 트랜지션이 기존과 동일하게 동작해야 한다.
- **FR-004**: 테마 전환 시 시스템은 사용자의 작업 상태(입력 중인 내용, 스크롤 위치, 선택된 내비 항목, 열린 글)를 보존해야 한다.
- **FR-005**: 토글은 키보드만으로 접근·조작할 수 있어야 하며, 보조 기술에 그 목적과 현재 상태를 전달해야 한다.

**테마 적용**

- **FR-006**: 시스템은 선택된 테마를 앱의 모든 화면(로그인, 스플래시, 업무, 마이 페이지)에 적용해야 한다.
- **FR-007**: 다크 테마는 모든 UI 요소 — 배경, 텍스트, 보더, 입력 필드, 버튼, 팝오버, 그림자, 빈 상태 — 를 어두운 팔레트로 표시해야 하며, 어떤 화면에도 라이트 테마 잔재가 남지 않아야 한다.
- **FR-008**: 다크 테마는 제품 정체성인 인디고 바이올렛 액센트와 각 색이 맡은 역할(강조·위험·성공·즐겨찾기)을 유지하되, 어두운 배경에서 읽히도록 값을 대응시켜야 한다.
- **FR-009**: 두 테마 모두에서 텍스트와 배경의 대비는 접근성 기준(본문 4.5:1, 큰 텍스트 및 UI 요소 3:1)을 충족해야 한다.

**유지 및 기본값**

- **FR-010**: 시스템은 사용자가 선택한 테마를 방문 간에 유지해야 한다.
- **FR-011**: 시스템은 테마 설정을 사용 중인 기기·브라우저 단위로 저장해야 한다. 설정은 로그인한 계정에 묶이지 않으므로, 같은 계정으로 다른 기기에서 접속하면 그 기기에 저장된 설정(없으면 FR-013)을 따른다.
- **FR-012**: 시스템은 앱이 처음 그려지는 시점부터 저장된 테마를 반영해야 하며, 다른 테마가 먼저 보였다가 바뀌는 깜빡임이 없어야 한다.
- **FR-013**: 저장된 설정이 없으면 시스템은 운영체제의 다크 모드 설정을 따라야 한다 — 운영체제가 다크면 다크로, 아니면 라이트로 표시한다.
- **FR-014**: 토글은 라이트와 다크 두 상태만 오가야 한다. "운영체제 따름"은 사용자가 고를 수 있는 상태가 아니라, 아직 선택이 없을 때 적용되는 초기 규칙이다.
- **FR-015**: 사용자가 토글로 테마를 선택하면 그 선택은 운영체제 설정보다 우선하며, 이후 운영체제 설정이 바뀌어도 유지되어야 한다.
- **FR-016**: 사용자가 아직 테마를 선택하지 않은 동안에는 운영체제 설정 변경을 앱에 반영해야 한다.
- **FR-017**: 저장된 값이 없거나 손상되었거나 저장소를 사용할 수 없는 경우, 시스템은 오류 없이 FR-013의 규칙으로 동작해야 하며 해당 세션 내 전환은 계속 가능해야 한다.
- **FR-018**: 마이 페이지의 "모든 데이터 초기화"는 글·프로필·세션을 지우되 테마 설정은 보존해야 한다. 초기화 직후 이동하는 로그인 화면도 기존 테마 그대로 표시되어야 한다.

### Key Entities

- **테마 환경설정 (Theme Preference)**: 사용자가 어떤 시각적 테마로 앱을 볼지에 대한 선택. 값은 라이트 또는 다크 둘 중 하나이며, "선택된 적 없음" 상태가 따로 존재한다. 선택된 적 없음일 때는 운영체제 설정으로 테마가 결정되고(FR-013), 사용자가 토글하는 순간 라이트 또는 다크로 확정되어 이후 운영체제 설정보다 우선한다(FR-015). 기기·브라우저마다 하나씩 존재하며 계정과 무관하게 방문 간 유지된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 토글을 조작한 뒤 화면 전체가 새 테마로 바뀌기까지 체감 지연이 없다(즉시 반응으로 인지되는 0.2초 이내).
- **SC-002**: 다크 테마를 선택한 사용자가 재방문했을 때 100% 다크 테마로 열리며, 잘못된 테마가 먼저 보이는 깜빡임이 0회다.
- **SC-003**: 두 테마 각각에서 텍스트·배경 조합의 100%가 접근성 대비 기준(본문 4.5:1, 큰 텍스트·UI 3:1)을 통과한다.
- **SC-004**: 다크 테마에서 앱의 4개 화면 전부를 둘러봤을 때 밝은 배경이 남아 있는 영역이 0곳이다.
- **SC-005**: 처음 보는 사용자가 안내 없이 5초 이내에 사이드바에서 테마 토글을 찾는다.
- **SC-006**: 마우스 없이 키보드만으로 토글에 도달해 테마를 전환할 수 있다.
- **SC-007**: 테마를 전환해도 작성 중이던 글 내용이 유실되는 사례가 0건이다.

## Assumptions

- **적용 범위**: 다크모드는 앱 전체(로그인·스플래시·업무·마이 페이지)에 적용한다. 사용자는 토글 위치만 지정했으나, 일부 화면만 어두운 앱은 결함으로 인식되기 때문이다.
- **토글 위치**: 토글은 사이드바에 둔다(사용자 명시). 이 앱에는 사이드바가 업무 화면과 마이 페이지 두 곳에 있으며, 둘 다에 토글을 둔다(2026-07-16 확인). 설정 화면인 마이 페이지에서 테마를 바꿀 수 없으면 사용자 기대에 어긋나기 때문이다. 사이드바가 없는 로그인 화면과 스플래시에는 토글을 노출하지 않으며 저장된 테마를 따르기만 한다.
- **디자인 시스템 연속성**: 다크 테마는 새로운 디자인 언어가 아니라 기존 색 역할의 다크 대응이다. 기존 토큰 체계와 액센트(`#6a5df0` 계열)의 정체성을 유지한다. 필요한 다크 값이 없으면 기존 스케일과 일관되게 추가하고 `DESIGN.md`에 동기화한다.
- **저장 범위 결정**: 테마를 계정이 아닌 기기 단위로 저장하기로 확정했다(FR-011). 로그인 전 화면에도 테마를 적용할 수 있고 서버 왕복이 없어 깜빡임(FR-012)을 막기 쉬우며, 헌법 원칙 IV(제로코스트·클라이언트 우선)와도 맞기 때문이다. 대가로 기기 간 동기화는 제공하지 않는다.
- **로그아웃 시 유지**: 테마 선택은 기기에 남으므로 로그아웃해도 초기화되지 않는다.
- **탭 간 실시간 동기화 제외**: 여러 탭이 열려 있을 때 한 탭의 전환이 다른 탭에 즉시 반영될 필요는 없다.
- **외부 이미지 미조정**: 글 커버의 외부 고양이 사진은 다크 테마에서도 원본 그대로 둔다.
- **범위 제외**: 테마별 전용 로고·일러스트, 인쇄용 스타일, 라이트/다크 외 추가 테마(세피아 등), 자동 시간대별 전환, 사용자 지정 색 커스터마이징, 기기 간 테마 동기화는 이번 범위가 아니다.
- **기존 사용자**: 이 기능 이전의 사용자는 저장된 테마 설정이 없으므로 FR-013의 운영체제 설정 규칙을 그대로 적용받는다.
