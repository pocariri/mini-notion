**Status**: Draft

**Input**: User description: "제목 입력창 위에 커버 이미지를 놓자. 이미지는 오픈API인 랜덤 고양이 사진 API를 사용할 거야. API 엔드포인트는 https://cataas.com/cat/cute 이거야. API가 로드되는 동안에는 로딩 스피너 대신 스켈레톤 UI로 처리해줘."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 글을 열면 제목 위에 고양이 커버가 보인다 (Priority: P1)

사용자가 워크스페이스에서 글을 열면, 제목 입력창 바로 위에 랜덤 고양이 사진이 커버 이미지로 표시된다. 노션의 페이지 커버처럼 에디터 상단을 장식해, 밋밋한 편집 화면에 개성과 재미를 더한다.

**Why this priority**: 이 기능의 핵심 가치다. 커버 이미지가 표시되지 않으면 기능 자체가 존재하지 않는 것과 같다. 이 스토리만 구현해도 완결된 MVP가 된다.

**Independent Test**: 워크스페이스에서 글 하나를 열고, 제목 입력창 위에 고양이 사진이 커버 형태로 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 워크스페이스에서 글을 선택하지 않은 상태, **When** 사용자가 글 하나를 열음, **Then** 제목 입력창 위 커버 영역에 랜덤 고양이 사진이 표시된다.
2. **Given** 한 글을 보고 있는 상태, **When** 사용자가 다른 글로 전환, **Then** 커버가 새로 로드되어 새 글의 커버 이미지가 표시된다.
3. **Given** 커버 이미지가 로드된 상태, **When** 사용자가 제목·내용을 입력, **Then** 커버는 그대로 유지되고 편집 도중 다시 로드되지 않는다.

---

### User Story 2 - 로딩 중에는 스켈레톤이 자리를 지킨다 (Priority: P2)

커버 이미지가 외부 서비스에서 로드되는 동안, 커버가 놓일 자리에 회전 스피너 대신 스켈레톤 UI(은은하게 일렁이는 플레이스홀더)가 표시된다. 이미지가 도착하면 스켈레톤이 이미지로 자연스럽게 교체된다.

**Why this priority**: 외부 이미지 로드는 체감 지연이 큰 구간이다. 스켈레톤이 최종 커버와 같은 자리를 미리 차지해 화면이 덜컹이지 않아야 하고, 사용자 요구사항에서 스피너 금지·스켈레톤 사용이 명시되었다.

**Independent Test**: 느린 네트워크 상태(또는 네트워크 스로틀링)에서 글을 열어, 이미지가 도착하기 전까지 커버 자리에 스켈레톤이 표시되고 스피너는 어디에도 나타나지 않는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 글을 여는 순간, **When** 커버 이미지가 아직 로드되지 않음, **Then** 커버 영역에 스켈레톤 UI가 표시되고 회전 스피너는 표시되지 않는다.
2. **Given** 스켈레톤이 표시된 상태, **When** 이미지 로드가 완료됨, **Then** 스켈레톤이 사라지고 같은 자리에 이미지가 표시되며 제목·본문의 위치가 밀리지 않는다.
3. **Given** 스켈레톤이 표시된 상태, **When** 사용자가 제목 입력창을 클릭해 입력, **Then** 로딩과 무관하게 제목 입력이 즉시 가능하다.

---

### User Story 3 - 이미지 로드에 실패해도 편집은 멀쩡하다 (Priority: P3)

외부 이미지 서비스 장애나 오프라인 등으로 커버 이미지를 가져오지 못한 경우, 커버 영역은 조용한 대체 상태(중립적인 빈 커버)로 전환되고, 글 편집 기능은 아무 영향 없이 그대로 동작한다.

**Why this priority**: 커버는 장식 요소이므로, 외부 서비스 실패가 핵심 기능(글 작성·편집)을 해쳐서는 안 된다. 발생 빈도는 낮지만 안정성의 기준선이다.

**Independent Test**: 네트워크를 차단한 상태(오프라인)에서 글을 열어, 스켈레톤이 무한히 지속되지 않고 대체 상태로 전환되며 제목·내용 편집이 정상 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 외부 이미지 서비스에 접근할 수 없는 상태, **When** 사용자가 글을 열음, **Then** 커버 영역이 스켈레톤에서 중립적인 대체 상태로 전환되고 오류 팝업·경고창은 표시되지 않는다.
2. **Given** 커버 로드가 실패한 상태, **When** 사용자가 제목·내용을 입력하고 다른 글로 전환, **Then** 모든 편집·전환 동작이 평소와 동일하게 동작한다.

---

### Edge Cases

- 이미지 로드가 매우 느릴 때 → 스켈레톤이 계속 표시되며, 그동안에도 제목·내용 편집은 차단되지 않는다.
- 글을 빠르게 연달아 전환할 때 → 이전 글에서 요청한 이미지가 늦게 도착하더라도 현재 열려 있는 글의 커버 상태를 덮어쓰지 않는다.
- 글을 선택하지 않은 빈 상태(환영 화면) → 에디터가 없으므로 커버도 스켈레톤도 표시되지 않는다.
- 휴지통의 읽기 전용 글을 열었을 때 → 조회는 가능하므로 커버도 동일하게 표시된다(로딩·실패 처리도 동일).
- 창 폭이 좁아질 때 → 커버가 에디터 영역 너비에 맞춰 줄어들고 가로 스크롤이나 레이아웃 깨짐이 발생하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 에디터에서 글을 열면 제목 입력창 바로 위에 커버 이미지 영역을 표시해야 한다.
- **FR-002**: 커버 이미지는 외부 랜덤 고양이 사진 오픈 API(`https://cataas.com/cat/cute`)에서 가져와야 한다.
- **FR-003**: 이미지가 로드되는 동안 커버 영역에는 스켈레톤 UI를 표시해야 하며, 회전 스피너를 사용해서는 안 된다.
- **FR-004**: 스켈레톤은 최종 커버 이미지와 동일한 크기·위치를 차지해, 로드 전후로 제목 입력창과 본문의 위치가 이동하지 않아야 한다.
- **FR-005**: 이미지 로드 실패 시 스켈레톤을 무한히 유지하지 않고 중립적인 대체 상태로 전환해야 하며, 오류가 편집 기능을 방해해서는 안 된다.
- **FR-006**: 커버 이미지 로딩은 제목·내용 입력을 차단하지 않아야 한다(로딩 중에도 즉시 편집 가능).
- **FR-007**: 다른 글로 전환하면 커버는 새 글 기준으로 다시 로드 과정을 거쳐야 한다(스켈레톤 → 이미지 또는 대체 상태).
- **FR-008**: 글을 빠르게 전환하는 경우에도, 늦게 도착한 이전 요청의 이미지가 현재 글의 커버를 덮어쓰지 않아야 한다.
- **FR-009**: 커버 이미지는 글에 저장하지 않으며, 글에 진입할 때마다 새 랜덤 이미지를 요청해야 한다. 같은 글을 다시 열면 다른 고양이가 나올 수 있다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 글을 열었을 때, 커버 영역이 처음부터 최종 크기로 자리를 차지해 이미지 로드 전후 제목·본문의 세로 위치 이동이 0이다.
- **SC-002**: 커버 로딩 구간에서 회전 스피너가 표시되는 경우가 0건이다(항상 스켈레톤으로 표시).
- **SC-003**: 커버 이미지 로드 성공·실패와 무관하게, 사용자는 글을 연 직후 즉시 제목 입력을 시작할 수 있다(로딩으로 인한 입력 대기 시간 0초).
- **SC-004**: 외부 이미지 서비스가 완전히 실패한 상황에서도 글 열기·편집·전환·저장 등 기존 기능의 성공률이 100% 유지된다.

## Assumptions

- 커버는 모든 글의 에디터 화면에 항상 표시되며, 사용자가 커버를 추가/제거/변경하는 기능(노션의 "커버 추가" 버튼류)은 이번 범위 밖이다.
- 글을 선택하지 않은 빈 상태(환영 화면)에는 커버를 표시하지 않는다.
- 외부 API(`https://cataas.com/cat/cute`)는 사용자가 지정한 외부 의존성으로, 무료 오픈 API이며 별도 인증·비용이 없다는 전제다(제로코스트 원칙 부합). 서비스 가용성은 보장되지 않으므로 실패 시 대체 상태로 처리한다.
- 커버의 구체적 높이·모서리·스켈레톤 질감 등 시각 세부는 기존 디자인 시스템(`DESIGN.md`) 토큰을 따르며, 계획(plan) 단계에서 확정한다.
