# Feature Specification: 자기소개 (Profile Introduction)

**Feature Branch**: `003-profile-introduction`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "자기소개 기능을 구현해줘. 마이페이지에는 현재 별명 기능만 존재해. 추가로 자기소개도 등록/수정/조회 가능하도록 만들어줘. DB에 introduction 컬럼 만들었으니까 연결하면 돼. 내가 만든 DB는 변경하면 안 돼."

## Clarifications

### Session 2026-07-16

- Q: 자기소개 최대 글자 수를 몇 자로 할까(별명은 20자)? → A: 150자 — 한두 문장짜리 짧은 소개. 입력란은 3줄 안팎으로 유지해 설정 화면에서 별명·이미지와의 시각적 균형을 지킨다.
- Q: 저장된 자기소개를 불러오지 못했을 때 어떻게 동작해야 하나? → A: 프로필 탭 폼 전체를 오류 상태로 두고 저장을 막으며 재시도 수단을 제공한다. 사용자가 보지 못한 값을 덮어쓰는 일을 원천 차단하는 대신, 불러오기 실패 시 별명·이미지 저장도 함께 막히는 기존 동작 변경을 감수한다(FR-019·FR-020).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 마이페이지에서 자기소개를 등록하고, 다시 열면 그대로 보인다 (Priority: P1)

사용자가 마이페이지의 프로필 탭에서 별명 아래에 있는 자기소개 입력란에 자신을 소개하는 글을 여러 줄로 적고 저장한다. 나중에 마이페이지를 다시 열면 저장했던 자기소개가 입력란에 그대로 채워져 있다. 지금까지 별명만 남길 수 있던 프로필에, 자신이 어떤 사람인지 설명할 공간이 생긴다.

**Why this priority**: 이 기능의 핵심 가치다. 등록과 조회가 되지 않으면 기능 자체가 존재하지 않는 것과 같다. 이 스토리만 구현해도 완결된 MVP가 된다.

**Independent Test**: 마이페이지 프로필 탭에서 자기소개를 입력하고 저장한 뒤, 페이지를 새로 고쳐 다시 들어가 같은 내용이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 자기소개를 한 번도 등록하지 않은 사용자가 마이페이지 프로필 탭에 있음, **When** 자기소개 입력란에 내용을 적고 저장을 누름, **Then** 저장 성공 안내가 표시되고 입력한 내용이 유지된다.
2. **Given** 자기소개를 저장한 상태, **When** 페이지를 새로 고치거나 다시 마이페이지에 진입, **Then** 저장했던 자기소개가 입력란에 그대로 채워져 있다.
3. **Given** 자기소개 입력란에 포커스가 있는 상태, **When** 사용자가 줄바꿈을 포함해 여러 줄을 입력, **Then** 줄바꿈이 그대로 입력되고 저장 후에도 유지된다.
4. **Given** 자기소개를 저장한 상태, **When** 로그아웃 후 같은 계정으로 다시 로그인해 마이페이지에 진입, **Then** 저장했던 자기소개가 그대로 표시된다.

---

### User Story 2 - 등록한 자기소개를 수정하거나 비울 수 있다 (Priority: P2)

이미 자기소개를 등록한 사용자가 내용을 고쳐 쓰거나, 전부 지워서 자기소개가 없는 상태로 되돌릴 수 있다. 자기소개는 선택 항목이므로, 비워 두는 것도 정상적인 선택지다.

**Why this priority**: 등록만 되고 되돌릴 수 없으면 사용자가 잘못 쓴 내용에 갇힌다. 다만 P1(등록·조회)이 동작해야 의미가 생기므로 그다음 순위다.

**Independent Test**: 자기소개가 등록된 상태에서 내용을 다른 문장으로 고쳐 저장하고 다시 조회해 반영을 확인한 뒤, 전부 지우고 저장해 빈 상태로 되돌아가는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 자기소개가 등록된 상태, **When** 내용을 다른 문장으로 고치고 저장, **Then** 새 내용이 저장되고 다시 열었을 때 새 내용이 표시된다.
2. **Given** 자기소개가 등록된 상태, **When** 내용을 전부 지우고 저장, **Then** 자기소개가 없는 상태로 저장되고 다시 열면 입력란이 비어 있으며 안내 문구(placeholder)가 보인다.
3. **Given** 자기소개가 비어 있는 상태, **When** 사용자가 아무것도 입력하지 않음, **Then** 저장 버튼은 비활성 상태로 남는다(변경 사항 없음).
4. **Given** 별명과 자기소개를 모두 고친 상태, **When** 저장을 한 번 누름, **Then** 두 값이 함께 저장되고 별명만 저장되거나 자기소개만 저장되는 일이 없다.
5. **Given** 자기소개가 등록된 상태, **When** 사용자가 계정 탭의 초기화를 실행, **Then** 별명·이미지와 함께 자기소개도 초기 상태로 지워진다.

---

### User Story 3 - 길이 제한과 오류 상황을 안전하게 안내받는다 (Priority: P3)

자기소개는 최대 150자까지 쓸 수 있고, 사용자는 현재 몇 글자를 썼는지 입력 중에 항상 확인할 수 있다. 저장이 실패하면 사용자가 쓴 내용이 사라지지 않고, 무엇이 잘못됐는지 안내받는다. 저장된 프로필을 불러오지 못한 경우에는 저장이 막히고 다시 시도할 수 있어, 보지 못한 내용을 모르는 사이에 덮어쓰지 않는다.

**Why this priority**: 없어도 등록·수정은 동작하지만, 제한을 모른 채 길게 쓰다 잘리거나 저장 실패로 글이 날아가면 신뢰를 잃는다. 별명이 이미 같은 방식(카운터·저장 실패 안내)을 제공하므로 일관성 기준선이다. 불러오기 실패 시 덮어쓰기 차단은 사용자가 인지할 수 없는 데이터 유실을 막는 안전장치다.

**Independent Test**: 150자에 가깝게 입력하며 카운터가 정확히 증가하는지, 초과 입력이 막히는지 확인하고, 저장이 실패하는 상황을 만들어 입력 내용이 유지된 채 오류 안내가 뜨는지 확인한다. 이어서 프로필 불러오기가 실패하는 상황을 만들어 저장이 막히고 재시도가 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 자기소개 입력란에 입력 중, **When** 글자를 입력하거나 지움, **Then** 현재 글자 수와 최대 글자 수(150)가 별명과 같은 형식으로 즉시 갱신되어 표시된다.
2. **Given** 자기소개가 150자에 도달한 상태, **When** 사용자가 글자를 더 입력하거나 긴 내용을 붙여넣기, **Then** 150자를 넘는 입력은 반영되지 않는다.
3. **Given** 자기소개를 수정한 상태, **When** 저장이 실패함, **Then** 오류 안내가 표시되고 사용자가 입력한 내용은 입력란에 그대로 남아 있으며, 저장된 자기소개는 이전 값 그대로 유지된다.
4. **Given** 저장이 진행 중인 상태, **When** 저장이 끝나기 전, **Then** 저장 버튼은 진행 중 상태로 표시되고 중복 저장이 발생하지 않는다.
5. **Given** 자기소개가 저장되어 있는 사용자, **When** 마이페이지 진입 시 프로필을 불러오지 못함, **Then** 프로필 탭 폼이 오류 상태로 표시되고 저장 버튼이 비활성화되며 다시 시도할 수 있는 수단이 제공된다.
6. **Given** 불러오기 실패로 폼이 오류 상태인 상황, **When** 사용자가 다시 시도해 불러오기가 성공함, **Then** 폼이 정상 상태로 돌아가고 저장되어 있던 별명·자기소개가 입력란에 채워지며 저장이 가능해진다.

---

### Edge Cases

- 자기소개를 한 번도 등록한 적 없는 사용자(값이 비어 있음) → 입력란은 빈 상태로 보이고 안내 문구(placeholder)가 표시되며, 오류로 취급하지 않는다.
- 공백·줄바꿈만 입력하고 저장 → 앞뒤 공백을 제거한 결과가 비어 있으므로 "자기소개 없음"으로 저장된다.
- 앞뒤에 공백을 포함해 입력 → 저장되는 값은 앞뒤 공백이 제거된 내용이다.
- 150자를 크게 넘는 내용을 붙여넣기 → 앞에서부터 150자까지만 입력되고 나머지는 잘린다(별명과 동일한 동작).
- 저장 중에 다른 탭(계정)으로 이동했다가 돌아옴 → 진행 중이던 저장 결과가 유실되거나 입력값이 뒤섞이지 않는다.
- 자기소개만 수정하고 별명·이미지는 그대로 → 별명·이미지 값이 의도치 않게 바뀌거나 지워지지 않는다.
- 로그인하지 않은 상태로 마이페이지 접근 → 기존 동작(로그인 화면으로 이동)과 동일하며, 자기소개 때문에 달라지지 않는다.
- 저장된 프로필(자기소개 포함)을 불러오지 못한 경우 → 프로필 탭 폼이 오류 상태로 표시되고 저장이 막히며, 사용자는 다시 시도할 수 있다. 사용자가 보지 못한 기존 자기소개를 덮어쓰는 일이 발생하지 않는다.
- 불러오기 실패 후 재시도가 성공한 경우 → 폼이 정상 상태로 돌아오고, 불러온 별명·자기소개가 입력란에 채워지며 저장이 다시 가능해진다.
- 창 폭이 좁아질 때 → 자기소개 입력란이 설정 화면 너비에 맞춰 줄어들고 가로 스크롤이나 레이아웃 깨짐이 발생하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 마이페이지 프로필 탭에 별명 입력란과 함께 자기소개 입력란을 제공해야 한다.
- **FR-002**: 자기소개 입력란은 줄바꿈을 포함한 여러 줄 입력을 지원해야 한다.
- **FR-003**: 자기소개는 최대 150자까지 입력할 수 있어야 하며, 이를 초과하는 입력은 반영되지 않아야 한다.
- **FR-004**: 자기소개 입력란에는 현재 글자 수와 최대 글자 수(150)를 별명과 동일한 형식으로 표시하고, 입력에 따라 즉시 갱신해야 한다.
- **FR-005**: 자기소개는 선택 항목이어야 한다. 비어 있어도 저장이 가능해야 하며, 비어 있다는 이유로 오류를 표시해서는 안 된다.
- **FR-006**: 자기소개를 저장할 때 앞뒤 공백을 제거한 값을 저장해야 한다.
- **FR-007**: 앞뒤 공백을 제거한 결과가 비어 있으면 "자기소개 없음" 상태로 저장해야 한다.
- **FR-008**: 마이페이지에 진입하면 이미 저장된 자기소개를 입력란에 채워 보여줘야 한다.
- **FR-009**: 자기소개만 변경한 경우에도 저장 버튼이 활성화되어야 한다.
- **FR-010**: 별명·프로필 이미지·자기소개를 함께 변경한 경우, 한 번의 저장으로 모두 반영되어야 하며 일부만 저장되어서는 안 된다.
- **FR-011**: 자기소개 저장에 성공하면 별명과 동일한 방식의 성공 안내를 표시해야 한다.
- **FR-012**: 자기소개 저장에 실패하면 별명과 동일한 방식의 오류 안내를 표시하고, 사용자가 입력한 내용을 입력란에 유지하며, 저장된 값은 이전 상태 그대로 두어야 한다.
- **FR-013**: 저장이 진행되는 동안 저장 버튼은 진행 중 상태로 표시되어 중복 저장을 막아야 한다.
- **FR-014**: 자기소개를 변경하는 저장이 별명·프로필 이미지 등 다른 프로필 값을 의도치 않게 변경하거나 지워서는 안 된다.
- **FR-015**: 계정 초기화 기능을 실행하면 자기소개도 다른 프로필 값과 함께 초기 상태로 지워져야 한다.
- **FR-016**: 자기소개는 본인만 조회하고 수정할 수 있어야 하며, 다른 사용자에게 노출되어서는 안 된다.
- **FR-017**: 자기소개는 기존에 준비된 저장소의 자기소개 항목에 보관해야 하며, 저장소 스키마를 변경해서는 안 된다.
- **FR-018**: 기존 별명·프로필 이미지의 등록·수정·조회 동작은 이번 변경으로 달라지지 않아야 한다. 단, 프로필 불러오기에 실패한 경우 저장이 막히는 것(FR-019)은 의도된 변경이며 이 항목의 예외다.
- **FR-019**: 마이페이지 진입 시 저장된 프로필을 불러오지 못하면, 프로필 탭의 입력 폼(별명·프로필 이미지·자기소개)을 오류 상태로 표시하고 저장을 막아야 하며, 사용자가 불러오기를 다시 시도할 수 있는 수단을 제공해야 한다.
- **FR-020**: 불러오기 실패로 저장이 막혀 있는 동안에는 저장된 프로필 값(별명·프로필 이미지·자기소개)이 어떤 방식으로도 변경되지 않아야 한다.
- **FR-021**: 불러오기 재시도가 성공하면 폼은 정상 상태로 돌아가고, 불러온 값이 입력란에 채워지며 저장이 다시 가능해져야 한다.

### Key Entities *(include if data involved)*

- **프로필(Profile)**: 로그인한 사용자 한 명을 나타내는 개인 정보 묶음. 사용자당 하나만 존재한다. 속성: 별명(표시 이름, 필수, 최대 20자), 이메일(로그인 계정에서 파생, 읽기 전용), 프로필 이미지(선택), **자기소개(선택, 여러 줄 평문, 최대 150자, 값이 없을 수 있음)**. 자기소개는 이번 기능에서 새로 다루는 속성이며, 저장소에는 이미 자리가 마련되어 있다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 자기소개를 저장한 뒤 페이지를 다시 열었을 때, 저장한 내용이 줄바꿈까지 100% 동일하게 표시된다.
- **SC-002**: 사용자는 마이페이지 진입 후 자기소개 입력란을 찾아 내용을 남기고 저장을 완료하기까지 별도 안내 없이 30초 이내에 끝낼 수 있다.
- **SC-003**: 150자를 초과한 자기소개가 저장되는 경우가 0건이다.
- **SC-004**: 저장에 실패했을 때 사용자가 입력한 내용이 유실되는 경우가 0건이다.
- **SC-005**: 자기소개 저장으로 인해 별명·프로필 이미지 값이 의도치 않게 변경되거나 지워지는 경우가 0건이다.
- **SC-006**: 프로필을 정상적으로 불러온 상태에서, 기존 별명 등록·수정·조회 흐름의 동작과 성공률이 100% 그대로 유지된다(회귀 0건). 불러오기 실패 시 저장을 막는 동작은 의도된 변경이므로 회귀에서 제외한다.
- **SC-007**: 이번 기능 구현으로 저장소 스키마를 변경하는 작업이 0건이다.
- **SC-008**: 저장된 프로필을 불러오지 못한 상태에서, 사용자가 보지 못한 기존 자기소개가 덮어써지는 경우가 0건이다.

## Assumptions

- **저장소 스키마는 변경하지 않는다.** 사용자가 이미 자기소개용 항목(`profile.introduction`)을 만들어 두었고, 본인 프로필을 조회·수정할 수 있는 권한 규칙도 이미 존재하므로 스키마·권한 변경 없이 연결만 하면 된다는 전제다(사용자 명시 제약).
- **자기소개는 본인만 본다.** 현재 제품에는 다른 사용자의 프로필을 보는 화면이 없고, 저장소 권한 규칙도 본인 프로필만 조회하도록 되어 있다. 다른 사용자에게 자기소개를 공개하려면 권한 규칙 변경이 필요한데 이는 위 제약에 어긋나므로, "조회"는 본인이 마이페이지에서 자신의 자기소개를 보는 것으로 한정한다.
- **별명과 같은 폼, 같은 저장 버튼을 쓴다.** 마이페이지 프로필 탭은 이미 별명·프로필 이미지를 하나의 저장 버튼으로 함께 저장한다. 자기소개도 같은 폼에 추가해 한 번에 저장하며, 자기소개 전용 저장 버튼은 두지 않는다.
- **자기소개는 선택 항목이다.** 별명은 비울 수 없지만(표시 이름이므로), 자기소개는 남기지 않는 것이 정상적인 상태다.
- **평문이다.** 마크다운·리치 텍스트 서식, 이미지·링크 임베드는 이번 범위 밖이다. 줄바꿈만 지원한다.
- **입력 제한 방식은 별명과 동일하다.** 150자 초과는 오류 메시지 대신 입력 자체를 막는 방식(하드 캡)으로 처리하고, 카운터로 남은 분량을 알린다. 저장소의 자기소개 항목 자체에는 길이 제약이 없으므로, 150자 제한은 저장소 변경 없이 앱에서 적용한다.
- **글자 수 세는 기준은 기존 카운터와 같다.** 공백·줄바꿈을 포함해 세며, 별명 카운터 및 본문 글자 수 세기와 동일한 규칙을 따른다.
- **프로필 이미지·이메일·계정 탭의 동작은 이번 범위 밖**이며 변경하지 않는다. 단 하나의 예외로, 프로필 불러오기 실패 시에는 폼 전체가 오류 상태가 되어 별명·이미지 저장도 함께 막힌다(FR-019). 자기소개만 선택적으로 잠그는 대신 폼 전체를 막는 쪽을 택한 이유는, 불러오기에 실패하면 별명·이미지 역시 신뢰할 수 없는 값(로그인 계정의 기본값)이 표시되어 같은 덮어쓰기 위험을 갖기 때문이다.
- 입력란 높이·여백·카운터 위치 등 시각 세부는 기존 디자인 시스템(`DESIGN.md`)의 토큰과 기존 여러 줄 입력 패턴을 따르며, 계획(plan) 단계에서 확정한다.
